# AgoraFold.BlazorWasm Architecture

## Summary

`AgoraFold.BlazorWasm` + `AgoraFold.BlazorWasm.Client`, the Blazor WebAssembly front-end variant — a hosted-WASM project pair (`dotnet new blazor --interactivity WebAssembly` shape). Goals and feature scope match every other variant (`design/project-spec.md`); this doc covers the two-project split, the hand-rolled JSON API + SPA-style auth design (chosen over the stock `dotnet new blazor --auth Individual` scaffold), and the render-mode gotchas that came from mixing a static host shell with interactive WASM islands. Ports: `http://localhost:5161`, `https://localhost:7137`.

Unlike the future React/Svelte/Angular/SolidJS variants (which will consume the existing `AgoraFold.WebApi`), this variant gets its **own backend** referencing `AgoraFold.Core` directly, like every other Blazor/Razor variant.

## Project split

- **`AgoraFold.BlazorWasm`** (host): owns `AddDbContext`/`AddIdentity`/`AddAgoraFoldCore()`/`Configure<ListingImageStorageOptions>` (copied verbatim from `AgoraFold.Htmx`/`AgoraFold.BlazorServer`), plus a hand-rolled JSON API — `Controllers/` (`AccountController`, `AntiforgeryController`, `CategoriesController`, `ListingsController`, `ConversationsController`) and `Filters/` (`AgoraFoldExceptionFilter`, `ValidateCsrfTokenAttribute`) — ported near-verbatim from `AgoraFold.WebApi`. Serves the compiled WASM client, `wwwroot/uploads/listings`, and the root `App.razor`/`Routes.razor` shell.
- **`AgoraFold.BlazorWasm.Client`**: the browser-side WASM assembly. Holds every interactive page (`Pages/`), the DTOs (`Api/Dto/`, mirroring `AgoraFold.WebApi/Models/*` 1:1), a thin `ApiClient` HTTP wrapper (`Api/ApiClient.cs`, analogous to `AgoraFold.Vue`'s `client.ts`), and a custom `CookieAuthenticationStateProvider` (`Auth/`). **No reference to `AgoraFold.Core`** — EF Core/Npgsql aren't browser-wasm-safe, and DB credentials have no business anywhere near the client assembly anyway. The host references the Client project (required so `AddInteractiveWebAssemblyComponents()` can discover its root components), so the host's controllers map Core entities directly to/from the Client's DTO records instead of duplicating them.

## Auth: hand-rolled JSON API, not the stock Identity scaffold

A deliberate choice over `dotnet new blazor --auth Individual`, which keeps Login/Register/Manage as static server-rendered pages and everything else as WASM — a pattern BlazorServer already demonstrates in spirit (static SSR auth + interactive elsewhere). This variant instead goes fully SPA-style: `AccountController` (`api/account/register|login|logout|me`) is a near-verbatim port of `AgoraFold.WebApi`'s, and Login/Register are ordinary interactive WASM `EditForm`s posting via `HttpClient` — no page reloads anywhere, no static/interactive render-mode split for the auth pages themselves.

- **Same-origin, no CORS**: unlike `AgoraFold.WebApi` (consumed cross-origin by the Vue dev server), the WASM client is served by the same Kestrel process it calls, so cookies flow automatically on same-origin `fetch` — no `AllowCredentials()`/`BrowserRequestCredentials.Include` needed anywhere.
- **CSRF**: same mechanism as `AgoraFold.WebApi` — `AddAntiforgery(o => o.HeaderName = "X-CSRF-TOKEN")`, `AntiforgeryController`, `ValidateCsrfTokenAttribute` calling `IAntiforgery.ValidateRequestAsync` directly. Client-side, `Api/CsrfHandler.cs` (a `DelegatingHandler`) fetches a **fresh** token before every mutating request via a second, undecorated `"ApiRaw"` named `HttpClient` (so the token fetch doesn't loop back through itself) — never cached, since a token issued before login/logout is rejected once the identity changes (same gotcha `AgoraFold.Vue`'s `client.ts` documents).
- **`CookieAuthenticationStateProvider`** (`Auth/`): calls `GET api/account/me` to build a `ClaimsPrincipal`, caches the result, and exposes `MarkAuthenticated`/`MarkLoggedOut` so Login/Register/Logout can update auth state directly from the response they already have instead of re-fetching `/me`. `CurrentUserAccessor` ports over from BlazorServer unchanged (it only wraps `AuthenticationStateProvider.GetAuthenticationStateAsync()`, already render-mode-agnostic) — except `ClaimsPrincipal.FindFirstValue` had to become `FindFirst(...)?.Value`, since the `FindFirstValue` extension lives in an ASP.NET Core hosting package not meant for the browser-wasm target.
- **Exception mapping**: `ApiClient` (`Api/ApiClient.cs`) translates HTTP 404/403/400 into `ApiNotFoundException`/`ApiForbiddenException`/`ApiValidationException` (`Api/ApiExceptions.cs`), so pages catch these the same way BlazorServer's pages catch Core's `NotFoundException`/`ForbiddenException`/`ValidationException` — `DomainErrorAlert` and `ErrorList` port over with only the exception types changed.

## Render modes: why the layout couldn't just be interactive

The biggest non-obvious fight in this variant. The natural first instinct — set one `@rendermode InteractiveWebAssemblyRenderMode(prerender: false)` on `<Routes>` in `App.razor` so the whole app (layout, nav, every page) shares a single interactive root and DI scope — **doesn't work**:

1. `Routes.razor` lives in the Host project's assembly, which is never shipped to the browser. An interactive WASM root component must be part of the Client assembly, so `<Routes @rendermode="...">` fails at runtime with *"Root component type ... could not be found in the assembly"*.
2. Moving `MainLayout` into the Client project and giving it the rendermode directly *(so `Routes`/`AuthorizeRouteView` stay static and just instantiate it)* fails differently: `InvalidOperationException: Cannot pass the parameter 'Body' to component 'MainLayout' ... because the parameter is of the delegate type RenderFragment, which is arbitrary code and cannot be serialized.` A static parent handing an interactive root its `@Body` content is exactly the kind of non-serializable parameter the static-to-interactive boundary rejects.

The fix, and the actually-supported pattern (matching what BlazorServer and the stock templates already do): **each page declares its own `@rendermode` individually** (`@rendermode @(new InteractiveWebAssemblyRenderMode(prerender: false))`, mirroring BlazorServer's per-page `@rendermode InteractiveServer`), and `MainLayout` stays in the Host project, fully static. The one piece of the layout that needs interactivity — the `AuthorizeView`-driven nav plus a clickable Logout button — is pulled out into a small separate island, `Client/Layout/AuthNav.razor`, embedded in the static `MainLayout` via `<AuthNav />`. Since `AuthNav` takes no parameters at all, it crosses the static→interactive boundary cleanly. Because it shares the same `InteractiveWebAssembly` render mode as whichever page is showing, both participate in the same single WASM runtime/DI scope for that page load — `AuthNav`'s `LogoutAsync` calling `CookieAuthenticationStateProvider.MarkLoggedOut()` is visible to the current page's own `AuthorizeView`/`[Authorize]` checks immediately, no full reload needed.

Prerendering is disabled (`prerender: false`) on every page: without it, each interactive component's lifecycle methods run twice — once server-side during prerender (no browser, no cookie jar, so authenticated API calls fail or return anonymous results) and once again after WASM actually boots. Disabling it is a simpler fix than plumbing cookie forwarding through the prerender pass, consistent with this project's general preference for idiomatic-per-variant code over defensive complexity (see BlazorServer's `IDbContextFactory` rejection rationale in `design/blazor-server-architecture.md`).

`RedirectToLogin.razor` (Host, static) drops BlazorServer's `forceLoad: true` — `NavigationManager.NavigateTo` already produces a real HTTP redirect when invoked from a non-interactive/static context (no circuit to force an exit from), and a plain client-side navigation is correct once WASM has booted, so `forceLoad` was never needed here.

## Image upload

Same shape as `AgoraFold.WebApi`: `IBrowserFile.OpenReadStream(maxAllowedSize)` buffered into a `MemoryStream`, wrapped as `ListingFileUpload(FileName, ContentType, Content)`, then posted as `MultipartFormDataContent` to `POST api/listings` (create, multipart-bound `ListingCreateForm`) or `POST api/listings/{id}/images` (`AddImagesForm`) — both host-only model classes (`Models/Listings/`) since `IFormFile` is a server-only ASP.NET Core type that has no business in the Client project. Validation (magic-byte sniffing, 5 MB/file, 8 images/listing) is unchanged, still enforced server-side in `IListingImageService`.

**Gotcha**: `ListingApiClient.CreateAsync`/`AddImagesAsync` originally read `using var content = BuildListingFormContent(...); return api.PostFormAsync(...);` — a non-`async` method whose `using` block disposes `content` (and the `StreamContent`/image streams it wraps) the instant the method returns the *Task*, not after that Task completes. Since `PostFormAsync` itself returns almost immediately (the actual HTTP send is still in flight), the multipart content got disposed out from under an in-progress `fetch()` call, surfacing as `ObjectDisposedException: MultipartFormDataContent` — caught by browser-testing the Create-with-image flow, not by the compiler. Fixed by making both methods properly `async`/`await` the inner call so the `using` scope actually outlives the request.

## Messaging

Same as every other variant: reply re-fetches the full thread, no push/polling. `ConversationThreadResponse` (unchanged from `AgoraFold.WebApi`) has no other-party display name field — `AgoraFold.Vue`'s own thread view already omits that detail for the same reason, so this variant matches the established API contract rather than inventing a new field.

## Gotchas

1. Root component discovery: an interactive WASM root **must** live in the Client project's assembly — the Host project's own Razor components (`Routes.razor`, `MainLayout.razor`, `NotFound.razor`) can never carry `@rendermode` themselves. See "Render modes" above.
2. A static parent can't hand an interactive root a `RenderFragment` parameter (e.g. `@Body`) — this is what actually blocks "make the whole layout one interactive root" from working, not a DI-scope-sharing concern.
3. `ClaimsPrincipal.FindFirstValue(...)` (the extension method) isn't available to the Client project; use `FindFirst(...)?.Value` instead.
4. `IHttpClientFactory`/`AddHttpClient` isn't included by default in a Blazor WASM client project — needs an explicit `Microsoft.Extensions.Http` package reference. Likewise `Microsoft.AspNetCore.Components.Authorization` needs its own explicit package reference (WASM's `Microsoft.AspNetCore.Components.WebAssembly` package doesn't pull it in transitively the way the Server hosting model does).
5. `using var content = ...; return SomeAsyncCall(content);` inside a non-`async` method disposes `content` before the async call actually finishes — the `MultipartFormDataContent` bug above. Any method with a `using` around something an inner `Task` still needs must itself be `async`/`await` the call, not just forward the `Task`.
